# HeAgent 设计说明

> 本文档说明 HeAgent **想要完成的主要功能与设计意图**（产品 / 设计视角）。代码如何实现见 [`frame.md`](frame.md)；如何使用见仓库根 [`README.md`](../README.md)。

## 一、定位与愿景

HeAgent 是一个**自学习 AI Agent 框架**，灵感来自 NousResearch 的 Hermes Agent。它不是 Hermes 的功能复刻，而是提取自主 Agent 的核心设计模式——**学习闭环、多模型适配、工具编排、记忆系统**——用更简洁、模块化的代码重新实现。

首要目标不是功能广度，而是：**通过重新实现这些模式来深入理解自主 Agent 的架构原理**，并产出一份额「活的架构教科书」——每一行代码都能回答「为什么这样设计」。

- **代码可读性 + 插件化** 优先于功能完整性。
- 目标用户：想理解 Agent 内部机制、实践插件化架构的开发者。

## 二、要解决的问题（设计动机）

以 Hermes 等框架为对照，HeAgent 想用更清晰的方式表达 Agent 的核心（解决「理解 gap」）：

- **巨型类耦合** → 模块解耦：核心循环、工具、记忆、Provider 各自独立。
- **学习曲线陡峭** → 自改进逻辑集中可见（技能提炼 / 记忆 / 审查）。
- **扩展不直观** → 声明式插件注册（加工具 / Provider 不改核心代码）。

## 三、主要功能

1. **Provider 抽象层** — 统一 OpenAI / Anthropic / DeepSeek / 智谱AI 等兼容接口；三层容错（重试 → 多 key 轮换 → 跨 Provider 回退）。
2. **工具调用引擎** — `@tool` 声明式注册（自动生成 JSON Schema）+ `asyncio.gather` 并行执行 + 安全护栏（shell 黑/白名单）。
3. **自学习闭环** — 从任务执行中提炼技能（`SKILL.md`，自动匹配注入）、跨会话事实记忆（`MEMORY.md`，关键词去重）、用户画像（`USER.md`）、SOUL 人格系统（全局 + 项目两级）。
4. **上下文管理** — 超阈值自动压缩、迭代 / token 预算控制、会话持久化。
5. **子 Agent 委派** — 生成隔离上下文的子 Agent，并行处理子任务并回收结果。
6. **MCP 协议集成** — 将外部 MCP server 的工具作为工具来源接入（声明式 `.mcp.json`）。
7. **Cron 定时调度** — asyncio 后台调度器，支持一次性与循环任务（交互模式自动启用）。*注：超出最初 V1 规划，后增量加入。*
8. **运行时治理（`engine/`）** — 策略门控的工具执行（准入 / 审批 / 沙箱裁决）、运行上下文与快照 checkpoint、幂等租约、事件可观测性。*epic 外 P0 增量。*

## 四、明确非目标（边界）

- 不做消息平台网关（Telegram / Discord / Slack）。
- 不做 TUI 终端界面（仅程序化 API + 简单 CLI）。
- 不做浏览器自动化 / 语音识别 / 图像生成。
- **安全非真正边界**：`SafetyGuard` / `path_safety` / `engine` sandbox 均非 OS 级隔离，不可在不可信内容下裸跑（须配容器 / firejail）。详见 [`frame.md`](frame.md) 安全声明。

## 五、设计原则与成功标准

- 核心循环类目标 <1,500 行；每个模块 <400 行；函数 <50 行。
- 插件化验证：加新工具 / Provider 不需要改核心代码。
- 能跑通完整 Agent 循环：用户输入 → LLM 推理 → 工具调用 → 结果返回 → 循环；技能提炼与记忆跨会话可演示；子 Agent 委派可工作。
