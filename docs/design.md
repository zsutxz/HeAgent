# HeAgent 设计说明

> 这份文档解释 HeAgent 想解决什么问题、刻意不解决什么问题，以及做设计取舍时遵循的原则。代码级实现细节见 [frame.md](frame.md)。

## 定位

HeAgent 是一个面向学习、实验和原型验证的 AI Agent 框架。项目目标不是尽快堆出最多功能，而是把自主 Agent 的关键机制拆开、做清楚、做成可阅读的 Python 代码。

它重点覆盖以下主题：

- LLM Provider 适配与故障转移
- 工具注册与调用编排
- 长短期记忆与人格注入
- 子 Agent 角色化协作
- 运行时治理、审批和可观测性
- MCP 工具接入

## 面向对象

更适合以下读者或使用者：

- 想理解 Agent 内部执行链路的开发者
- 想基于现有框架继续扩展工具、Provider、策略层的人
- 想做本地实验和架构验证，而不是直接拿去上线的人

## 设计动机

这个项目试图解决的不是“如何把所有 Agent 功能塞进一个仓库”，而是以下几类更基础的问题：

- 核心循环能否保持清晰，而不是被巨型类吞掉。
- Provider、工具、记忆、上下文、运行时治理能否解耦。
- 扩展新工具或新 Provider 时，能否尽量少改核心逻辑。
- 运行时状态、子任务和恢复点能否被结构化记录，而不是散落在消息里。

## 当前能力范围

HeAgent 当前已经实现并暴露给使用者的能力主要有：

1. Provider 抽象层
   支持 OpenAI、Anthropic，以及 DeepSeek 等兼容 OpenAI 协议的服务（DeepSeek 经 `OpenAIProvider` + `base_url` 接入，非独立 provider 类）；支持重试、key 池轮换、跨 Provider 回退，以及运行时经 `/model` 在多个 vendor 间手动/自动切换（`SwitchableProvider`）。
2. 工具调用系统
   支持 `@tool` 注册、schema 生成、并行工具执行、shell/file/search 等内置工具。
3. 记忆与人格
   支持技能、事实、用户画像、全局/项目级 `SOUL.md`。
4. 上下文管理
   支持上下文文件加载、压缩、窗口重置、会话持久化。
5. 子 Agent
   支持委派、并行执行以及 `planner`、`coder`、`tester`、`supervisor` 四个内置角色。
6. MCP
   支持通过 `.mcp.json` 注册外部 MCP server，接入 Tools（工具桥接 + 写操作治理）、Resources（按需发现/读取）、Prompts（slash 命令调度）三原语。
7. Cron
   支持在交互模式下启动后台调度器。
8. 运行时治理
   支持 `PolicyEngine`、`ToolExecutor`、`RunStore`、`ExecutionLedger`、事件发布。

## 明确非目标

当前阶段不把下面这些内容当作项目承诺：

- 完整的 Web API 或多租户服务端产品
- 可直接上线的安全边界
- 浏览器自动化、语音、图像、多端 UI 等外围能力
- 一套统一而成熟的部署产品形态

尤其需要明确：

- `SafetyGuard` 不是沙箱。
- 外部 MCP server 不是受信执行环境。
- “可运行”不等于“适合生产环境”。

## 设计原则

- 学习价值优先于功能堆砌。
- 模块边界优先于临时方便。
- 默认使用声明式扩展点，而不是到处写特殊分支。
- 运行时状态尽量结构化、可恢复、可观测。
- 文档描述应以代码事实为准，避免把规划稿当成现状。

## 成功标准

如果这个项目的设计是成功的，应该满足：

- 新增工具或 Provider 时，不需要重写 `AgentLoop`。
- 阅读 `src/heagent/` 可以顺着模块边界理解系统。
- 一次运行的上下文、工具执行和子任务结果能被追踪。
- 用户可以从 `README` 上手，从 `frame.md` 理解实现。

## 当前成熟度

截至当前仓库状态，HeAgent 更准确的描述是：

- 一个可运行的 CLI / Python 库
- 一个持续演进中的 Agent 架构实验场
- 一个比“生产代理服务”更适合拿来阅读、验证和扩展的项目

如果后续演进到真正的服务化部署、可信沙箱、稳定 API，再单独拆分新的设计文档更合适。
