# PRD: HeAgent

## 0. 文档目的

本文档定义 HeAgent（自改进 AI Agent 核心框架）的产品需求。面向读者：作者本人（开发者 + PM）和下游 BMAD 工作流（架构设计、Epic 拆分）。FR 全局编号，便于架构和 Story 引用。

上游输入：[产品简报](../brief-HeAgent-2026-05-23/brief.md)。

## 1. 愿景

HeAgent 是一个精简的、可自改进的 AI Agent 核心框架。它从 Hermes Agent 提取精华设计模式——学习闭环、多模型适配、工具编排——用更清晰、模块化的代码重新实现。每一行代码都回答"为什么这样设计"，而不是"这个功能也塞进来吧"。

项目目标不是功能覆盖，而是**架构理解**：通过亲手构建来掌握自主 Agent 的核心机制。

## 2. 目标用户

### 2.1 主要角色

**作者本人**——一个对 AI Agent 内部机制好奇的开发者，通过构建来学习。

### 2.2 任务目标

- 理解自主 Agent 如何从经验中学习（技能提炼 + 记忆管理）
- 掌握多模型 Provider 适配的工程模式（抽象层 + 回退链）
- 实践插件化架构设计（声明式注册、零耦合扩展）
- 产出可运行、可演示、可扩展的 Agent 框架

## 3. 术语表

- **Provider** — LLM API 适配器（如 OpenAI、Anthropic），实现统一接口，可插拔替换
- **Tool** — Agent 可调用的能力单元（终端执行、文件操作等），通过声明式注册
- **Skill** — 从任务执行中提炼的可复用操作模式，以 SKILL.md 格式持久化
- **Memory** — 跨会话的事实记忆（MEMORY.md）和用户画像（USER.md）
- **Iteration Budget** — 线程安全的迭代计数器，防止单次 Agent 运行的无限工具调用循环
- **SubAgent** — 由父 Agent 生成的隔离子实例，有独立的上下文和迭代预算
- **Context Window** — 模型的最大 token 容量；上下文压缩在接近此限制时自动触发
- **Fallback Chain** — Provider 失败时的有序备用列表；凭证轮换在同一 Provider 的多个 API Key 间切换
- **MCP** — Model Context Protocol，外部工具服务器的标准协议；HeAgent 通过 MCP 发现和调用外部工具

## 4. 功能需求

### 4.1 Provider 抽象层

统一多模型 API 的调用接口。每个 Provider 实现相同协议，核心循环无需关心底层差异。支持回退链和凭证轮换，确保单 Provider 故障不中断服务。

**功能需求：**

#### FR-1: Provider 接口定义

系统提供统一的 `BaseProvider` 协议，所有 Provider 实现必须遵循。协议包含：发送消息、流式发送、解析工具调用、获取模型元数据。

**后果（可测试）：**
- 新 Provider 实现只需继承 `BaseProvider` 并实现抽象方法，无需修改核心循环代码
- 所有 Provider 返回统一的消息格式（`ProviderResponse` Pydantic model）

#### FR-2: OpenAI Provider 实现

系统实现 OpenAI 兼容 API 的 Provider（支持 OpenAI 官方端点和兼容端点如 LM Studio）。

**后果（可测试）：**
- 可通过 OpenAI Provider 发送消息并接收完整响应
- 支持流式响应（SSE）
- 支持 function calling / tool_use 格式解析

#### FR-3: Anthropic Provider 实现

系统实现 Anthropic Claude API 的 Provider。

**后果（可测试）：**
- 可通过 Anthropic Provider 发送消息并接收完整响应
- 支持流式响应
- 支持 tool_use 格式解析
- 支持提示词缓存（cache_control 断点注入）`[ASSUMPTION: V1 先支持基础缓存，不做复杂的多断点策略]`

#### FR-4: Provider 回退链

系统维护有序的 Provider 列表，当前 Provider 失败时自动切换到下一个。

**后果（可测试）：**
- 当主 Provider 返回速率限制（429）或服务不可用（503）时，自动激活下一个 Provider
- 回退触发时记录日志（Provider 名称、错误类型、切换目标）
- 所有 Provider 均不可用时，向调用方返回明确错误

#### FR-5: 凭证轮换

同一 Provider 支持配置多个 API Key，配额耗尽时自动切换。

**后果（可测试）：**
- 单个 Key 触发速率限制时，自动切换到下一个 Key
- Key 池耗尽时，触发 Provider 级别的回退（FR-4）

### 4.2 工具调用引擎

声明式工具注册 + 并行/顺序执行调度。工具是 Agent 的"手脚"——所有与外部世界的交互都通过工具完成。

**功能需求：**

#### FR-6: 声明式工具注册

开发者通过装饰器或注册函数声明工具，系统自动提取工具名称、描述、参数 schema 并提供给 LLM。

**后果（可测试）：**
- 使用 `@tool` 装饰器注册的函数自动出现在 LLM 的可用工具列表中
- 添加新工具不需要修改核心循环或工具调度代码
- 工具参数 schema 自动从函数签名和类型注解生成

#### FR-7: 工具并行执行

当 LLM 返回多个无依赖关系的工具调用时，系统并行执行。

**后果（可测试）：**
- 多个工具调用在 `ThreadPoolExecutor` 中并行执行
- 所有结果收集完毕后才提交给 LLM 进入下一轮
- 单个工具执行失败不影响其他工具的并行执行

#### FR-8: 安全护栏

系统在执行工具前进行安全检查，拦截潜在危险操作。

**后果（可测试）：**
- 终端工具检测到危险命令（`rm -rf`、`format`、`dd` 等）时，暂停执行并请求用户确认
- 用户可配置白名单模式（仅允许列表内命令）或黑名单模式（阻止列表内命令）
- 拦截事件记录日志

#### FR-9: 内置工具集

V1 提供以下内置工具：

**终端执行工具** — 执行 shell 命令，返回 stdout/stderr/exit code。
**文件操作工具** — 读取、写入、搜索、编辑文件。
**网页搜索工具** — 通过 API 执行网页搜索，返回结构化结果。

**后果（可测试）：**
- 每个内置工具通过 FR-6 的声明式注册，可独立启用/禁用
- 终端工具有超时控制（默认 120 秒）

### 4.3 自学习系统

Agent 的核心差异化能力——从每次任务执行中提炼可复用的操作模式，跨会话记住用户偏好。

**功能需求：**

#### FR-10: 技能提炼

Agent 在完成一轮工具调用后，后台分析执行模式，将成功的操作序列提炼为 Skill。

**后果（可测试）：**
- 每轮对话结束后，系统自动评估是否产生了可提炼的操作模式
- 提炼的 Skill 以 SKILL.md 格式集中保存到统一目录（如 `.heagent/skills/`）
- 后续对话中，匹配的 Skill 自动注入系统提示词

#### FR-11: 事实记忆管理

系统维护 MEMORY.md 文件，记录跨会话的用户偏好、项目事实和关键决策。

**后果（可测试）：**
- Agent 可在对话中主动保存事实到 MEMORY.md
- 新会话启动时自动加载 MEMORY.md 内容到上下文
- 记忆条目通过关键词匹配去重，避免重复积累

#### FR-12: 用户画像

系统维护 USER.md 文件，记录用户的技术背景、偏好风格和交互习惯。

**后果（可测试）：**
- Agent 可在对话中更新 USER.md 的用户画像条目
- 新会话启动时自动加载 USER.md 到系统提示词
- 用户画像影响 Agent 的回复风格和技术深度

### 4.4 上下文管理

控制 token 消耗，防止单次会话耗尽模型上下文窗口。

**功能需求：**

#### FR-13: 自动上下文压缩

当对话 token 接近模型上下文窗口阈值时，系统自动压缩历史消息。

**后果（可测试）：**
- 压缩使用主模型调用总结中间对话轮次（后续可配置独立辅助模型）
- 压缩保留最近的 N 轮对话不压缩（保留活跃上下文）
- 压缩触发阈值可配置（默认 80% 上下文窗口）

#### FR-14: 迭代预算控制

每次 Agent 运行有最大迭代次数限制，防止单次任务的无限工具调用循环。

**后果（可测试）：**
- 每次运行开始时分配可配置的迭代预算（默认 50 次）
- 每次工具调用消耗一次迭代，达到上限时 Agent 必须输出最终响应
- 子 Agent 共享或独立分配预算（可配置）

#### FR-15: 会话持久化

对话历史可保存和恢复，支持跨会话继续。

**后果（可测试）：**
- 对话历史以 JSON 格式持久化
- 可通过 session ID 恢复之前的对话上下文
- 会话文件包含完整的消息历史和工具调用记录

### 4.5 子 Agent 委派

父 Agent 可生成隔离的子 Agent 并行处理子任务，结果回传汇总。

**功能需求：**

#### FR-16: 子 Agent 生成与执行

Agent 可通过工具调用委派任务给子 Agent 实例。

**后果（可测试）：**
- 子 Agent 有独立的对话上下文和迭代预算
- 子 Agent 默认继承父 Agent 的全部工具集
- 子 Agent 执行完毕后返回结果给父 Agent

#### FR-17: 并行子 Agent 编排

多个子 Agent 可并行执行。

**后果（可测试）：**
- 多个子 Agent 在独立线程中并行运行
- 所有子 Agent 结果收集完毕后汇总给父 Agent
- 单个子 Agent 失败不阻塞其他子 Agent

### 4.6 MCP 协议集成

通过 Model Context Protocol 连接外部工具服务器，作为工具来源的补充。

**功能需求：**

#### FR-18: MCP 工具发现与调用

系统可连接 MCP 服务器，发现其提供的工具并注册为可用工具。

**后果（可测试）：**
- 配置 MCP 服务器后，其工具自动出现在 LLM 可用工具列表中
- MCP 工具调用与内置工具使用相同的执行流程（安全护栏、超时控制）
- MCP 连接失败时降级（跳过该 MCP 的工具），不阻塞 Agent 运行

### 4.7 错误处理

**功能需求：**

#### FR-19: 错误分类与差异化重试

系统自动分类 API 错误并根据类型选择重试策略。

**后果（可测试）：**
- 速率限制错误 → 触发凭证轮换（FR-5）或 Provider 回退（FR-4）
- 认证错误 → 跳过当前 Provider，尝试回退链中的下一个
- 瞬态错误（网络超时、服务过载） → 指数退避重试（最多 3 次）
- 非瞬态错误 → 立即返回错误给调用方

## 5. 明确非目标

- **不是** Hermes 的功能复刻——不追求功能对等
- **不做**消息平台网关（Telegram / Discord / Slack / WhatsApp / Signal / Email）
- **不做** Cron 定时调度
- **不做** TUI 终端界面（仅程序化 API + 简单 CLI）
- **不做**浏览器自动化工具
- **不做**语音识别 / 图像生成
- **不做**多用户 / 多租户支持
- **不做** Web 服务端（FastAPI / HTTP 接口）

## 6. MVP 范围

### 6.1 MVP 包含

- FR-1 ~ FR-3：Provider 接口 + OpenAI + Anthropic 实现
- FR-4 ~ FR-5：回退链 + 凭证轮换
- FR-6 ~ FR-9：工具注册 + 调度 + 安全护栏 + 内置工具（终端、文件、搜索）
- FR-10 ~ FR-12：技能提炼 + 记忆管理 + 用户画像
- FR-13 ~ FR-15：上下文压缩 + 迭代预算 + 会话持久化
- FR-16 ~ FR-17：子 Agent 委派 + 并行编排
- FR-19：错误分类与重试

### 6.2 MVP 后延后

- FR-18（MCP 集成）— 优先完成核心闭环，MCP 作为扩展机制验证 `[NOTE FOR PM: MCP 是插件化架构的关键验证点，建议在核心闭环稳定后尽快实现]`
- Gemini / 本地模型 Provider — OpenAI + Anthropic 足以验证架构
- 提示词缓存策略优化 — V1 只做基础缓存

## 7. 成功指标

**主要指标：**

- **SM-1**：Agent 核心循环可端到端运行 — 用户输入 → LLM 推理 → 工具调用 → 结果返回 → 循环继续。验证 FR-1, FR-6。
- **SM-2**：技能提炼闭环可演示 — 完成任务后自动生成 SKILL.md，后续对话中复用。验证 FR-10。
- **SM-3**：插件化零耦合验证 — 添加新工具或新 Provider 不修改核心循环代码。验证 FR-1, FR-6。
- **SM-4**：代码可读性达标 — 核心循环模块 <1,500 行，单个模块 <400 行。

**次要指标：**

- **SM-5**：Provider 回退链在模拟故障下自动切换。验证 FR-4, FR-5。
- **SM-6**：子 Agent 可并行执行并回收结果。验证 FR-16, FR-17。

**反指标（不优化）：**

- **SM-C1**：功能数量 — 不以功能覆盖率为目标，宁缺毋滥。
- **SM-C2**：性能吞吐量 — 单 Agent 单会话场景，不做并发优化。
